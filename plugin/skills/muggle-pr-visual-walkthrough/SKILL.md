# Muggle PR Visual Walkthrough

Posts a visual walkthrough of Muggle AI E2E acceptance test results to the current branch's open PR. Each test case is linked to its detail page on the Muggle AI dashboard, so PR reviewers can click through to see step-by-step screenshots and action scripts — not just a pass/fail flag.

This is the canonical "attach results to PR" workflow shared by `muggle-test`, `muggle-test-feature-local`, and `muggle-do`. Those skills invoke this one at the end of a test run; it can also be invoked directly if the user asks to post existing results to a PR.

## When to use

- After `muggle-test` or `muggle-test-feature-local` finishes a local run and publishes results
- After `muggle-do` completes its E2E acceptance stage
- When the user asks "post my test results to the PR", "share the walkthrough on GitHub", or similar
- When the user wants reviewers to see screenshots of user flows rather than read log output

## Required input (must be in conversation context)

This skill does not call Muggle tools to fetch data — the caller must already have run the tests and published the results. Before invoking, the conversation must contain:

- **`projectId`** — the Muggle AI project ID, used to build dashboard URLs
- **Per test case:**
  - `title` — human-readable test case name
  - `testCaseId` — for the per-test dashboard link
  - `status` — `PASSED` or `FAILED`
  - `steps` — step count
  - `duration` — e.g. `12.3s`
  - `viewUrl` — returned by `muggle-local-publish-test-script` (local mode) or derived from the dashboard URL format (remote mode)
  - On failure: the failing step, error message, and local artifact / screenshot paths if available

If any of this data is missing, stop and tell the caller what's missing — do not guess or fabricate.

## Step 1: Find the PR

```bash
gh pr view --json number,url,title 2>/dev/null
```

Branch → three cases:

- **PR exists** → go to Step 2, post as a comment
- **No PR exists** → use `AskQuestion`:
  - "Create a new PR with the visual walkthrough in the body"
  - "Skip posting"
- **`gh` not installed or not authenticated** → tell the user, suggest `gh auth login`, and skip

## Step 2: Build the walkthrough comment body

Construct a markdown comment. Link each test case title to its detail page on the Muggle AI dashboard (`modal=script-details`, not `modal=details`), so reviewers get one-click access to the screenshots.

```markdown
## 🧪 Muggle AI — Visual Walkthrough

**X passed / Y failed** | [View all on Muggle AI](https://www.muggle-ai.com/muggleTestV0/dashboard/projects/{projectId}/runs)

| Test Case | Status | Details |
|-----------|--------|---------|
| [Login with valid creds](https://www.muggle-ai.com/muggleTestV0/dashboard/projects/{projectId}/scripts?modal=script-details&testCaseId={testCaseId}) | ✅ PASSED | 8 steps, 12.3s |
| [Login with invalid creds](https://www.muggle-ai.com/muggleTestV0/dashboard/projects/{projectId}/scripts?modal=script-details&testCaseId={testCaseId}) | ✅ PASSED | 6 steps, 9.1s |
| [Checkout flow](https://www.muggle-ai.com/muggleTestV0/dashboard/projects/{projectId}/scripts?modal=script-details&testCaseId={testCaseId}) | ❌ FAILED | Step 7: "Click checkout button" — element not found |

<details>
<summary>Failed test details</summary>

### Checkout flow
- **Failed at**: Step 7 — "Click checkout button"
- **Error**: Element not found
- **Local artifacts**: `~/.muggle-ai/sessions/{runId}/`
- **Screenshots**: `~/.muggle-ai/sessions/{runId}/screenshots/`

</details>

---
*Generated by [Muggle AI](https://www.muggle-ai.com) — click any test case above to see the step-by-step visual walkthrough*
```

Only include the `<details>` block when at least one test failed. Omit the "Local artifacts" / "Screenshots" lines for remote-mode runs where there are no local paths.

## Step 3: Post to GitHub

### If a PR already exists — add as a comment

```bash
gh pr comment {pr-number} --body "$(cat <<'EOF'
{the walkthrough comment body from Step 2}
EOF
)"
```

### If creating a new PR

Include the walkthrough section in the PR body alongside the usual Summary / Test Plan sections. Use `gh pr create` with a HEREDOC body. Do not create a PR without explicit user confirmation from Step 1.

## Step 4: Confirm to the user

> "Visual walkthrough posted to PR #{number}. Reviewers can click any test case link to see the step-by-step screenshots on the Muggle AI dashboard."

Include the PR URL.

## Tool Reference

| Phase | Tool |
|:------|:-----|
| Find PR | `gh pr view` |
| Post comment | `gh pr comment` |
| Create PR with walkthrough | `gh pr create` |
| User confirmation | `AskQuestion` |

## Guardrails

- **Never invent data** — if `projectId`, `testCaseId`, `viewUrl`, or a per-test-case status is missing from the caller's context, stop and report what's missing. Do not fabricate links or fill in placeholders.
- **Never create a PR without confirmation** — if there's no existing PR, ask the user before calling `gh pr create`.
- **Use `modal=script-details`** in dashboard URLs, not `modal=details` — the latter is a known-wrong format.
- **Only include the `<details>` failure block when there are failures** — don't add an empty block for all-passing runs.
- **Omit local-artifact paths for remote-mode runs** — they don't exist on the user's machine.
- **Don't run tests** — this skill only reports existing results. If the caller hasn't run tests yet, redirect them to `muggle-test` or `muggle-test-feature-local`.
